### Why We See "11 PyATS Jobs Per Device"

**Root Cause:** The job file generated by `JobGenerator.generate_device_centric_job()` contains a Python loop:

```python
# Generated job file content (job_generator.py:146-155)
for test_file in TEST_FILES:  # TEST_FILES has 11 items
    run(
        testscript=test_file,
        taskid=f"{hostname}_{test_name}",
        max_runtime=3600,
        testbed=runtime.testbed
    )
```

**Consequence:**
- 1 OS subprocess spawned per device ✅
- Inside that subprocess: 11 `run()` calls ❌
- Each `run()` creates a `multiprocessing.Process` (PyATS Task)
- 11 Tasks × 11s overhead = **121 seconds wasted**

## PyATS Framework Limitations (From Source Code Analysis)

### Critical Finding: Each run() ALWAYS spawns subprocess

**Evidence from PyATS source** (`/Users/oboehmer/Documents/Test-Automation/pyats-repo/pkgs/easypy-pkg/src/pyats/easypy/tasks.py`):

```python
# Line 182-191
class Task(multiprocessing.Process):
    '''Task class
    
    Task class represents the actual task/testscript being executed through a
    child process. All tasks within easypy are encapsulated in its own forked
    process from easypy main program.
    '''

# Line 153-179
def run(self, *args, max_runtime=None, **kwargs):
    '''run api
    
    Shortcut function to start a Task(), wait for it to finish, and return
    the result to the caller.
    '''
    task = Task(*args, **kwargs)  # Creates multiprocessing.Process
    task.start()                  # Forks a new subprocess
    task.wait(max_runtime)
    return task.result
```

**Conclusion:** 
- ✅ Confirmed: Each `pyats.easypy.run()` call **MUST spawn a subprocess**
- ❌ No built-in batching mechanism exists in PyATS
- ⚠️ This is by design for test isolation

## Test Execution Flow (Per Test File)

### 1. PyATS Subprocess Initialization (~3-5 seconds)

When `pyats.easypy.run(testscript=file, ...)` is called in the job file:

**What happens:**
```
1. Python interpreter starts new subprocess
2. Import PyATS framework modules
3. Load plugins (ProgressReporterPlugin)
4. Initialize aetest framework
5. Load testscript Python file
6. Import test dependencies (nac_test_pyats_common, jmespath, etc.)
```

**Estimated Time:** 3-5 seconds per subprocess

### 2. Test Class Setup Phase (~2-3 seconds)

**File:** `nac_test/pyats_core/common/base_test.py` (NACTestBase.setup, lines 179-213)

**Steps:**

## Identified Bottlenecks

### High Impact (>1 second per test)

#### 1. **PyATS Subprocess Initialization (3-5s per test)**
**Problem:** Each `run(testscript=...)` spawns new Python subprocess  
**Impact:** 3-5 seconds × 11 tests = **33-55 seconds per device**  
**Solution:** Run all tests in same subprocess (batch tests into single testscript)

#### 2. **Data Model File I/O (1-2s per test)**
**Problem:** Each test reads `merged_data_model_test_variables.yaml` from disk  
**File:** base_test.py line 186: `self.data_model = self.load_data_model()`  
**Impact:** 1-2 seconds × 11 tests = **11-22 seconds per device**  
**Solution:** 
- Cache data model in memory (class variable or module-level)
- Pass data model via job runtime instead of re-reading file

#### 3. **TestResultCollector Initialization (0.5-1s per test)**
**Problem:** Creates new directory + JSON file per test  
**File:** base_test.py lines 215-276  
**Operations:**
- Generate unique test ID
- Create `html_report_data_temp/<test_id>/` directory
- Initialize `<test_id>_results.json` file

**Impact:** 0.5-1 second × 11 tests = **5.5-11 seconds per device**  
**Solution:** Reuse result collector across tests (singleton pattern)

### Medium Impact (0.5-1 second per test)

#### 4. **Genie Parser Overhead (0.5-1s per test)**
**Problem:** Parsing IOS-XE command output is CPU-intensive  
**File:** ssh_base_test.py line 254: `self.testbed_device.parse(command, output=output)`  
**Impact:** 0.5-1 second × 11 tests = **5.5-11 seconds per device**  
**Solution:**
- Cache parsed results (if same command across tests)
- Use simpler regex parsing for known output formats
- Profile Genie parser to identify slow parsers

#### 5. **Result Serialization (0.5-1s per test)**
**Problem:** Writing JSON results to disk per test  
**Impact:** 0.5-1 second × 11 tests = **5.5-11 seconds per device**  
**Solution:** 
- Batch result writes (write once at end of all tests)
- Use in-memory buffer instead of disk I/O per test

### Low Impact (<0.5 second per test)

#### 6. **Command Cache Overhead**
**Problem:** CommandCache instance created per test  
**File:** ssh_base_test.py line 222: `self.command_cache = CommandCache(hostname)`  
**Impact:** Minimal (cache is per-hostname, should be shared)  
**Solution:** Verify cache is actually shared across tests

#### 7. **Environment Variable Parsing**
**Problem:** Each test re-reads and parses DEVICE_INFO from env var  
**File:** ssh_base_test.py line 96-106  
**Impact:** <0.1 second per test  
**Solution:** Parse once and pass via runtime

## Optimization Strategies

### Option A: Consolidate Test Files (RECOMMENDED)
- **Action:** Merge 11 test files into 1 file with 11 testcase classes.
- **Impact:** Reduces 11 `run()` calls to 1 `run()` call.
- **Savings:** 121s → 22s (eliminates 10 subprocess spawns = ~99 seconds saved).
- **Implementation:** EASY (1-2 hours).
- **Risk:** LOW (standard PyATS pattern).

### Option B: Add runtime.max_workers
- **Action:** Add `runtime.max_workers = 5` to job file.
- **Impact:** Runs tests in parallel (5 at a time).
- **Savings:** 121s → ~50s (tests run in 3 batches: 5+5+1).
- **Implementation:** Does NOT reduce subprocess count, just parallelizes them.
- **Risk:** LOW (API tests already use this).

### Option C: Combination (BEST)
- **Action:** Consolidate tests AND add `max_workers`.
- **Impact:** Single subprocess with parallel test execution inside.
- **Savings:** 121s → 15-20s (Meets 30-second target!).
- **Risk:** MEDIUM (need to ensure test independence).

## Recommended Implementation Plan

### Phase 1: Quick Wins (Target: 60-80 seconds per device)

1. **Cache data model** (save 10-20s)
   - Modify `load_data_model()` to use module-level cache
   - Test: Run scale test, verify only 1 file read per device

2. **Reuse result collectors** (save 5-10s)
   - Attach result collector to runtime/parent
   - Test: Verify only 1 directory created per device

3. **Batch result writes** (save 3-5s)
   - Buffer results in memory
   - Write once at end of all tests

**Expected Result:** 115s → 80s per device (~30% faster)

### Phase 2: Major Refactor (Target: 30-40 seconds per device)

4. **Combine tests into single testscript** (save 30-50s)
   - Merge 11 test files into 1 file with 11 test methods
   - Test: Verify only 1 subprocess spawned per device
   - Risk: Some test isolation lost

**Expected Result:** 80s → 30s per device (2.5× faster, meets target!)

### Phase 3: Advanced Optimization (Target: <30 seconds)

5. **Profile and optimize Genie parsers** (save 5-10s)
   - Use `cProfile` to identify slow parsers
   - Replace with regex where possible
   - Cache parsed outputs

6. **Minimize I/O operations**
   - Use in-memory buffers instead of files
   - Defer disk writes to end of job

**Expected Result:** 30s → 20-25s per device (additional 20-30% faster)
